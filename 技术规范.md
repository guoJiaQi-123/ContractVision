# 二、项目概述
## 2.1 项目背景
传统企业销售合同管理存在**数据分散、统计困难、分析滞后、可视化程度低、履约跟踪难**等问题。本项目基于Python技术栈构建一站式合同管理与数据可视化平台，实现合同全生命周期管理、多维度分析、智能预警、权限管控、报表导出等功能。

## 2.2 项目目标
1. 实现企业销售合同数据**集中化、标准化、可视化**管理。
2. 提供多维度数据分析图表，支持经营决策。
3. 实现精细化角色权限与数据权限管控。
4. 支持合同履约、回款、到期、预警自动化。
5. 系统稳定、安全、易扩展、易部署。
6. 满足毕业设计规范与企业实际使用场景。

## 2.3 核心业务范围
- 用户认证与权限管理
- 销售合同全生命周期管理（增删改查/导入导出/变更/作废/审批）
- 合同履约、回款、付款、续签、预警管理
- 多维度数据可视化分析（业绩、区域、产品、客户、团队）
- 自定义报表与多格式导出
- 系统管理、日志审计、数据备份

# 三、技术架构选型
## 3.1 整体架构（前后端分离）
采用 **前后端分离 + MVC + 三层架构**
- 前端：视图层与交互层
- 后端：控制层、业务逻辑层、数据访问层
- 数据库：持久化存储
- 缓存：提升查询性能

## 3.2 后端技术栈
| 技术类别 | 选型 | 版本要求 | 选型理由 |
|----------|------|----------|----------|
| 开发语言 | Python | ≥3.9 | 语法简洁、数据处理库丰富 |
| Web框架 | Django + Django REST Framework | ≥4.2 | 成熟稳定、自带Admin、权限体系完善 |
| 数据处理 | Pandas / NumPy | 最新版 | Excel处理、统计分析、数据计算 |
| 机器学习 | Scikit-learn | 最新版 | 销售预测、客户价值分析 |
| 认证授权 | JWT + Session | 标准 | 无状态认证、安全可靠 |
| 任务调度 | Celery | 最新版 | 自动预警、自动备份、定时任务 |
| 文档生成 | ReportLab / XlsxWriter | 最新版 | PDF/Excel报表导出 |

## 3.3 前端技术栈
| 技术类别 | 选型 |
|----------|------|
| 核心框架 | Vue 2 / Vue 3 |
| UI组件 | Element UI |
| 可视化库 | ECharts |
| 请求库 | Axios |
| 构建工具 | Vue CLI / Vite |

## 3.4 数据库与中间件
- 主数据库：MySQL 8.0+
- 缓存：Redis（用于登录状态、高频数据）
- 版本控制：Git
- 部署：Python虚拟环境 + Nginx + Gunicorn

## 3.5 架构优势
1. Django生态成熟，开发效率高，适合毕业设计快速交付
2. 前后端分离便于并行开发、独立部署、独立扩展
3. ECharts可视化能力强，满足企业数据分析需求
4. Redis缓存提升海量数据查询速度
5. 技术栈学习成本低、社区资源丰富

# 四、核心业务模块技术实现方案
## 4.1 用户认证与权限模块
1. **认证实现**
   - 采用JWT令牌认证
   - 密码使用PBKDF2加密存储
   - 登录失败次数限制（防暴力破解）
2. **权限实现**
   - 角色：管理员、操作员、查看用户
   - 功能权限：菜单级控制
   - 数据权限：行级过滤（区域/部门/业务员）
3. **技术点**
   - Django权限系统 + 自定义中间件
   - 登录日志、IP记录、异常登录告警

## 4.2 合同数据管理模块
1. **合同CRUD**
   - Django ORM 实现数据操作
   - 字段校验：合同编号唯一、金额非负、日期逻辑校验
2. **批量导入/导出**
   - 导入：Pandas读取Excel → 数据清洗 → 校验 → 入库
   - 导出：QuerySet → XlsxWriter生成Excel文件
3. **合同变更/作废/终止**
   - 逻辑删除，保留历史记录
   - 操作日志全记录

## 4.3 合同履约与预警模块
1. **履约跟踪**
   - 进度条百分比计算
   - 节点状态：未开始、进行中、已完成、逾期
2. **自动预警**
   - Celery定时任务
   - 每日扫描合同：到期、回款逾期、交付逾期
   - 站内消息 + 列表高亮标记

## 4.4 数据可视化模块
1. **数据统计**
   - 按时间/区域/产品/客户/团队分组统计
   - 同比/环比/占比/趋势计算
2. **图表渲染**
   - 后端返回标准JSON数据
   - 前端ECharts渲染：折线图、柱状图、饼图、雷达图、地图
3. **驾驶舱**
   - 自定义组件拖拽布局
   - 图表联动与下钻分析

## 4.5 报表模块
1. **动态报表**
   - 用户选择维度、指标、时间
   - 后端动态生成SQL/ORM查询
2. **导出格式**
   - Excel：XlsxWriter
   - PDF：ReportLab

## 4.6 系统日志与审计模块
- 中间件全局拦截请求
- 记录：操作人、IP、时间、类型、内容、前后数据对比
- 支持导出与条件查询

# 五、API 设计规范
## 5.1 基础规范
1. **请求域名**
   - `/api/模块名/版本/接口名`
   - 示例：`/api/contract/v1/list`
2. **请求方式**
   - 查询：GET
   - 新增：POST
   - 修改：PUT
   - 删除：DELETE
3. **状态码统一**
   - 200：成功
   - 400：参数错误
   - 401：未登录
   - 403：无权限
   - 500：服务器错误

## 5.2 请求参数规范
- 分页：`page=1&size=10`
- 排序：`sort=amount&order=desc`
- 时间：`start_time&end_time`
- 参数统一使用驼峰或下划线，**项目内保持一致**

## 5.3 响应格式统一
```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "list": [],
    "total": 100
  }
}
```

## 5.4 接口安全
- 必须带Token：`Authorization: Bearer token`
- 敏感接口限流
- 入参过滤XSS与SQL注入风险

## 5.5 接口文档
- 使用 **Swagger / DRF Docs** 自动生成
- 每个接口必须标注：功能、入参、出参、错误码

# 六、开发规范
## 6.1 代码风格指南
### 6.1.1 Python 编码规范
1. 遵循 **PEP8**
2. 缩进使用 **4个空格**
3. 文件名：小写+下划线，如 `contract_service.py`
4. 类名：大驼峰，如 `ContractService`
5. 函数/变量：小写+下划线，如 `get_contract_list`
6. 常量：全大写+下划线，如 `MAX_IMPORT_SIZE`

### 6.1.2 注释规范
- 类必须写文档字符串：功能、作者、日期
- 公共函数必须标注：参数、返回值、异常
- 复杂逻辑必须添加行注释
- 禁止无意义注释

### 6.1.3 项目结构规范
```
project/
├── apps/           # 业务模块
├── core/           # 核心工具
├── utils/          # 工具类
├── settings/       # 配置
├── static/         # 静态资源
└── templates/      # 模板
```

## 6.2 分支管理策略
### 6.2.1 分支命名
- `main`：生产环境代码（不可直接提交）
- `dev`：开发主分支
- `feature/xxx`：功能分支，如 `feature/contract-import`
- `bugfix/xxx`：修复分支
- `release/v1.0`：发布分支

### 6.2.2 合并规则
1. 功能开发 → 提交feature分支
2. 提交Pull Request
3. Code Review通过后合并到dev
4. 测试完成后合并到main发布

### 6.2.3 提交信息规范
格式：`类型: 内容描述`
- `feat: 新增合同批量导入功能`
- `fix: 修复权限判断错误`
- `refactor: 重构可视化统计逻辑`
- `docs: 更新技术规范文档`

## 6.3 代码审查流程（Code Review）
1. 开发者提交PR/MR
2. 审查人检查：
   - 代码规范
   - 逻辑正确性
   - 安全性（SQL注入、XSS、越权）
   - 异常处理
   - 性能问题
3. 审查通过 → 合并
4. 不通过 → 标注问题 → 重新修改提交

## 6.4 测试覆盖率标准
1. **单元测试**
   - 服务层、工具类必须编写单元测试
2. **接口测试**
   - 所有API必须通过接口测试
3. **覆盖率要求**
   - 核心业务代码 ≥ 80%
4. **测试工具**
   - pytest + django.test

# 七、非功能性需求保障与部署指南
## 7.1 非功能性技术保障
### 7.1.1 安全性
1. SQL注入防护：使用ORM/参数化查询
2. XSS防护：前端转义、后端过滤
3. 越权防护：行级权限+功能权限校验
4. 敏感信息加密：密码、密钥加密存储
5. 接口限流、IP黑名单

### 7.1.2 性能
1. 列表查询添加索引
2. 高频数据使用Redis缓存
3. 大数据分页优化
4. 导入导出异步化

### 7.1.3 兼容性
1. 浏览器：Chrome、Edge、Firefox最新版
2. 移动端：响应式适配
3. 分辨率：1920×1080及以上

### 7.1.4 可靠性
1. 定时自动备份
2. 异常捕获与日志记录
3. 服务崩溃自动重启

## 7.2 部署指南
### 7.2.1 环境要求
- Python 3.9+
- MySQL 8.0+
- Redis
- 虚拟环境：venv / conda

### 7.2.2 部署步骤
1. 拉取代码
2. 创建虚拟环境并安装依赖
3. 配置数据库、Redis、密钥
4. 执行数据库迁移
5. 收集静态文件
6. 使用Gunicorn启动服务
7. Nginx反向代理
8. 配置开机自启

### 7.2.3 部署命令
```bash
pip install -r requirements.txt
python manage.py migrate
python manage.py collectstatic
gunicorn project.wsgi:application --bind 0.0.0.0:8000
```
